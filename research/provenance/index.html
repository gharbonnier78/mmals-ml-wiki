Research record governance

From project conversations to a public research chronicle

Chats are useful working memory, but they are not automatically authoritative, complete, private-safe, or publication-ready. Diderot turns them into curated events under an explicit contract.

What is extracted

Research questions, version transitions, explicit hypotheses, negative results, branch launches, evidence decisions, repository publications, and changes in claim status.

What is not published automatically

Raw conversation transcripts, personal or organizational context, unverified claims, duplicated brainstorming, private links, credentials, or provisional wording that has not been checked against an artifact.

Four provenance classes

Artifact-grounded

Notebook, PDF, result ZIP, or release

The strongest chronicle source. It should include a stable identifier or repository mapping.

GitHub-verified

Public repository state

Used for repository existence, descriptions, and update status—not for scientific validity.

Curated project record

Conversation-derived event

A human-readable summary of a project decision whose raw source remains private or has no public artifact yet.

Planned

Proposal or pending branch

Visible as future work and never displayed as completed evidence.

Review lifecycle

1

Extract

Identify a proposed event from a conversation, artifact, release, or repository.

2

Normalize

Assign a date, branch, event type, concise summary, consequence, and provenance class.

3

Validate

Check names, versions, evidence status, public links, and whether the event belongs to core MMALS or an adjacent project.

4

Publish

Add the event to the chronicle and preserve it in version control.

5

Correct, never erase silently

Later evidence may qualify or supersede an event. The correction is added with a new date and an explicit relationship.
