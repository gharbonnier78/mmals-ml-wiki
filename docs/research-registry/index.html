Maintainer guide

Research registry governance

How to add events, branches, repositories, and publications without confusing activity, publicity, and scientific evidence.

Four connected registries

  • data/research-events.json: dated events, outcomes, branch, and provenance;
  • data/branches.json: lineage, maturity, claim boundary, repositories, and next step;
  • data/repositories.json: public, local, or planned repository state;
  • data/publications.json: separate GitHub, arXiv, HAL, DOI, and venue fields.

Do not infer evidence from activity

A recent commit, public repository, or polished PDF does not establish a scientific claim. Repository metadata may refresh automatically; branch maturity must be reviewed.

Adding an event

  1. Use an ISO date and a human-readable date label.
  2. Choose one primary branch and one event type.
  3. Write a neutral summary and a separate research consequence.
  4. Mark whether the provenance is a public artifact or a curated private project record.
  5. Check that no sensitive or irrelevant conversation detail is exposed.
